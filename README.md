# Triangdist Package 

## Math introduction aim to understand the mathematical background of the triangular distribution
The triangular distribution is a distribution for continuous random variables defined by three parameters: 
- **a**: the point where the distribution starts increasing 
- **c**: the point where the distribution starts decreasing
- **b**: the point where the distribution ends  

- **The probability density function (PDF) of the triangular distribution is defined as follows:**
$$
f(x) = \begin{cases}
0 & \text{if } x < a \\
\frac{2(x-a)}{(b-a)(c-a)} & \text{if } a \leq x < c \\
\frac{2(b-x)}{(b-a)(b-c)} & \text{if } c \leq x < b \\
0 & \text{if } x \geq b  
\end{cases}
$$

- **The cumulative distribution function (CDF) of the triangular distribution is defined as follows:**
$$
F(x) = \begin{cases}
0 & \text{if } x < a \\
\frac{(x-a)^2}{(b-a)(c-a)} & \text{if } a \leq x < c \\
1 - \frac{(b-x)^2}{(b-a)(b-c)} & \text{if } c \leq x < b \\
1 & \text{if } x \geq b 
\end{cases}
$$

## R Package Development

### Roxygen2
In R, functions inside a package need to be documented and exported explicitly.
Roxygen2 is the standard tool for this — instead of writing raw `.Rd` documentation 
files by hand (which use a horrible LaTeX-like syntax), you write special comments 
starting with `#'` directly above each function. The main tags are:

- `@param` — describes an input parameter
- `@return` — describes what the function returns
- `@export` — marks the function as public (without this, the function exists 
  inside the package but the user cannot call it directly)
- `@importFrom` — declares an external function your code uses (e.g. `@importFrom stats runif`)

When you run `devtools::document()`, roxygen2 reads these comments and 
automatically generates the `man/*.Rd` files and updates the `NAMESPACE` file.

### Package structure
An R package has a specific folder structure that R and CRAN expect:

- `R/` — all your `.R` source files with the actual functions
- `man/` — documentation files auto-generated by roxygen2, do not edit manually
- `tests/testthat/` — unit tests written with the `testthat` package
- `DESCRIPTION` — metadata: package name, version, author, dependencies (equivalent to `pyproject.toml` in Python)
- `NAMESPACE` — declares which functions are exported and which external functions are imported, auto-managed by roxygen2

### Useful commands
```r
devtools::document()   # regenerate NAMESPACE and man/ from roxygen2 comments
devtools::load_all()   # load the package locally for testing
devtools::check()      # full CRAN-style check: 0 errors, 0 warnings, 0 notes
```

### Unit Testing with testthat
R base has no built-in testing framework — the standard is the `testthat` package,
which is the pytest equivalent in R. It has no relation to roxygen2: roxygen2 is 
for documentation, testthat is for testing.

The basic structure of a test is:
```r
test_that("description of what you are testing", {
  expect_equal(dtriang(1, 0, 2, 1), 1)
})
```

`test_that` is the test block (equivalent to a `test_` function in pytest) and 
inside it you put `expect_*` calls which are the assertions. The most useful ones:

- `expect_equal(result, expected)` — checks equality with numeric tolerance
- `expect_error(expr, "message")` — checks that an error is thrown with that message
- `expect_true(condition)` — checks that something is TRUE

Test files must be named `test_*.R` and live in `tests/testthat/`. 
`devtools::check()` runs them all automatically.

To check code coverage (must be >95%) run:
```r
covr::report()
```

## Test Plan

To achieve >95% code coverage every branch of every `ifelse` must be exercised.
The tests are split into four files, one per function:

### `tests/testthat/test_dtriang.R`
- x below min → returns 0
- x above max → returns 0
- x between min and mode → left branch of PDF
- x between mode and max → right branch of PDF
- x exactly at min → returns 0
- x exactly at max → returns 0
- x exactly at mode → returns peak value `2/(max-min)`
- vectorized input → returns a vector of correct length
- invalid parameters: `min >= max` → error
- invalid parameters: `mode < min` → error
- invalid parameters: `mode > max` → error

### `tests/testthat/test_ptriang.R`
- q below min → returns 0
- q above max → returns 1
- q between min and mode → left branch of CDF
- q between mode and max → right branch of CDF
- q exactly at mode → returns `(mode-min)/(max-min)`
- q exactly at min → returns 0
- q exactly at max → returns 1
- vectorized input → returns a vector of correct length
- invalid parameters: `min >= max` → error
- invalid parameters: `mode < min` → error
- invalid parameters: `mode > max` → error

### `tests/testthat/test_qtriang.R`
- p = 0 → returns min
- p = 1 → returns max
- p = (mode-min)/(max-min) → returns mode
- p below threshold → left branch
- p above threshold → right branch
- vectorized input → returns a vector of correct length
- p < 0 → error
- p > 1 → error
- invalid parameters: `min >= max` → error
- invalid parameters: `mode < min` → error
- invalid parameters: `mode > max` → error

### `tests/testthat/test_rtriang.R`
- output length equals n
- all values are within [min, max]
- invalid parameters: `min >= max` → error
- invalid parameters: `mode < min` → error
- invalid parameters: `mode > max` → error
